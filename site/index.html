Prototype-to-firmware flow

The workflow mirrors how StopWatch apps are actually built: browser iteration first, then a focused hardware port, then host firmware integration through app submodules.

01
Prototype UI and interaction
Start from the HTML framework, tune the round-screen layout, buttons, pointer input, timing, audio, and vibration behavior in a browser.
02
Mirror confirmed behavior
Keep constants, state machines, result formats, and feedback gates aligned so the browser prototype remains the source of truth.
03
Migrate into Mooncake/LVGL
Extract the app repository, wire assets and lifecycle hooks, build with ESP-IDF, then flash and inspect device logs.

Skill package map

Inspired by agent skill pages that explain the directory contract first: a skill is not only prompt text, it also carries reusable references, scripts, and assets.

SKILL.md Workflow rules for StopWatch app design, Mooncake migration, validation, and repository handoff.
html-framework/ Standalone 466x466 browser prototype starter with keys, pointer input, feedback helpers, and embedded preview mode.
assets/html-framework/ The same framework bundled inside the skill so future app repos can be created from the approved template.
thirdparty/<App> Host firmware integration pattern: each app lives in its own repository and is consumed as a submodule.
Browser-firstFast visual iteration in plain HTML, no build step, and `file://` friendly previews.
Hardware-awareThe template uses the same circular display size, A/B button model, and feedback timing concepts as the device.
Evidence-drivenThe skill asks Codex to inspect repo state, build firmware, flash when requested, and avoid judging hardware behavior without logs.
